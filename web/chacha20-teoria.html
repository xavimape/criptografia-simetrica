ChaCha20-Poly1305 · RFC 8439

Teoria e internos

Como funciona ChaCha20-Poly1305

8 diapositivas: desde el diseno ARX hasta el esquema AEAD completo.

01 / 08

Que es ChaCha20-Poly1305?

ChaCha20-Poly1305 es un cifrado AEAD (Authenticated Encryption with Associated Data) estandarizado en el RFC 8439 (2018). Combina dos primitivas:

  • ChaCha20 — stream cipher ARX de 256 bits
  • Poly1305 — MAC de un solo uso basado en GF(2130−5)

Es el algoritmo predilecto cuando no hay hardware AES-NI disponible (moviles, dispositivos embebidos). Lo usan TLS 1.3, WireGuard, Signal, OpenSSH y muchos mas.

Key (256 bits) + Nonce (96 bits) | v ChaCha20 Block <-- counter (32 bits) | Keystream (64 B / bloque) | PT XOR KS = CT | Poly1305(AD || CT) = Tag (128 bits)
02 / 08

Diseno ARX: Add, Rotate, XOR

ChaCha20 usa solo tres operaciones sobre enteros de 32 bits:

\(a \leftarrow (a + b) \bmod 2^{32}\)  ·  \(a \leftarrow a \lll n\)  ·  \(a \leftarrow a \oplus b\)

ADD a = (a + b) mod 2^32 -- suma modular, sin carry ROTATE a = a <<< n -- rotacion izquierda de n bits XOR a = a XOR b -- or exclusivo

Por que ARX?

  • Sin tablas S-Box: inmune a ataques de cache-timing
  • Velocidad constante: sin ramas condicionales sobre datos secretos
  • Eficiente en CPU de 32 bits (ARM Cortex-M, RISC-V)
  • Facil de implementar sin side-channels

Disenado por Daniel J. Bernstein en 2008 como variante de Salsa20. El nombre viene de Chacha dance: las diagonales "bailan" respecto a las columnas.

03 / 08

Estado 4x4: la matriz de 512 bits

El estado interno es una matriz 4×4 de words de 32 bits (16 words = 512 bits totales), organizada en filas:

w0 w1 w2 w3 <-- Constantes: "expand 32-byte k" w4 w5 w6 w7 <-- Clave K[0..3] (128 bits) w8 w9 w10 w11 <-- Clave K[4..7] (128 bits) w12 w13 w14 w15 <-- Counter | Nonce[0..2]

Constantes (azul): "expa", "nd 3", "2-by", "te k" en ASCII — iguales para todos, no secretas.
Clave (verde): 256 bits del usuario en little-endian.
Counter (naranja): empieza en 0 para derivar la clave Poly1305, en 1 para cifrar. Permite acceso aleatorio a cualquier bloque de 64 bytes.
Nonce: 96 bits, unico por mensaje con la misma clave.

04 / 08

Quarter Round: la operacion fundamental

El Quarter Round (QR) transforma 4 words (a, b, c, d) con 8 operaciones ARX:

QUARTERROUND(a, b, c, d): a += b; d ^= a; d <<<= 16; c += d; b ^= c; b <<<= 12; a += b; d ^= a; d <<<= 8; c += d; b ^= c; b <<<= 7;

En cada QR los 4 words se mezclan completamente entre si. Las rotaciones (16, 12, 8, 7) estan elegidas para maxima difusion en el menor numero de rondas (analisis de Aumasson et al.).

Paralelismo: los 4 QR de cada ronda son independientes y pueden ejecutarse en paralelo (SIMD con SSE2/AVX2 da ~10 ciclos/byte).

05 / 08

Rondas: columnas y diagonales

ChaCha20 aplica 20 rondas (10 double-rounds). Cada double-round = ronda de columnas + ronda de diagonales:

COLUMNAS (las 4 columnas de la matriz): QR(w0, w4, w8, w12) QR(w1, w5, w9, w13) QR(w2, w6, w10, w14) QR(w3, w7, w11, w15) DIAGONALES (las 4 diagonales desplazadas): QR(w0, w5, w10, w15) <-- diagonal principal QR(w1, w6, w11, w12) <-- +1 QR(w2, w7, w8, w13) <-- +2 QR(w3, w4, w9, w14) <-- +3 x 10 = 20 rondas totales

Al mezclar columnas y luego diagonales, cada word afecta a todas las demas en 2 double-rounds. En 10 double-rounds la difusion es total. Salsa20 usaba filas y columnas; ChaCha20 usa columnas y diagonales para mayor resistencia con menos rondas.

06 / 08

Generacion del keystream y cifrado

Para producir el bloque i del keystream (64 bytes):

1. state = InitMatrix(key, nonce, counter=i) 2. working = state 3. for round in 1..20: COLUMNROUND(working) DIAGONALROUND(working) 4. keystream_block = (working + state) mod 2^32 -- suma word a word 5. return serialize_LE(keystream_block) Cifrado: CT_i = PT_i XOR keystream_i

La suma final (paso 4) hace que la funcion sea no invertible: aunque alguien obtenga el keystream, no puede recuperar el estado intermedio para derivar la clave.
El counter permite cifrar bloques en orden arbitrario (acceso aleatorio), a diferencia de CBC o CTR con carry.

07 / 08

Poly1305: el MAC de un solo uso

Poly1305 evalua un polinomio sobre GF(2130−5), un primo de Mersenne cercano a 2130:

$$acc_i = \bigl(acc_{i-1} + m_i + 2^{128}\bigr) \cdot r \pmod{2^{130} - 5}$$ $$tag = (acc + s) \bmod 2^{128}$$

Clave one-time: (r, s) ambos de 16 bytes r = poly1305_key[0..15] clamped a 106 bits efectivos s = poly1305_key[16..31] libre Mensaje = pad16(AD) || pad16(CT) || len(AD) || len(CT) acc = 0 for each 16-byte block m_i: acc = (acc + m_i + 2^128) * r mod (2^130 - 5) tag = (acc + s) mod 2^128

El valor (r, s) se deriva del bloque 0 del keystream (counter=0), garantizando que sea unico por mensaje. Si se reutiliza (r,s) con dos mensajes distintos, un atacante puede recuperar r. Poly1305 es seguro en el modelo UF-CMA cuando la clave es de un solo uso.

08 / 08

Esquema AEAD completo

El esquema AEAD garantiza confidencialidad y autenticidad:

Cifrado: (r, s) = ChaCha20Block(key, nonce, counter=0)[0..31] CT = ChaCha20(key, nonce, counter=1) XOR PT tag = Poly1305(r, s, AD, CT) output = CT || tag Descifrado: (r, s) = ChaCha20Block(key, nonce, counter=0)[0..31] tag' = Poly1305(r, s, AD, CT) if tag' != tag: ABORT -- rechazar sin descifrar PT = ChaCha20(key, nonce, counter=1) XOR CT

El AD (Associated Data) se autentica pero no se cifra — util para cabeceras de red o ID de sesion.
Nonce nunca debe repetirse con la misma clave: aunque no rompe la confidencialidad como en AES-GCM, compromete la autenticidad.
Limite: 232×64 bytes = 256 GiB por nonce.
