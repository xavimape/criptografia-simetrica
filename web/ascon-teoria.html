¿Qué es ASCON-128a?

What is ASCON-128a?

┌─────────────────────────────────────────────────────────┐ │ ASCON-128a — Cifrado Autenticado con Datos Asociados │ │ (Authenticated Encryption with AD) │ ├─────────────────────────────────────────────────────────┤ │ Entradas : Clave (128 bits) + Nonce (128 bits) │ │ + Datos Asociados (variable, sin cifrar) │ │ + Plaintext (variable) │ │ │ │ Salidas : Ciphertext (igual longitud que Plaintext) │ │ + Tag de autenticación (128 bits) │ └─────────────────────────────────────────────────────────┘
AEAD: Cifra el plaintext Y autentica tanto el ciphertext como los datos asociados (metadatos visibles pero protegidos contra alteración).
Estándar NIST LWC 2023: Seleccionado por el NIST como estándar de criptografía ligera. Ganador de CAESAR 2019 entre 57 candidatos. Diseñado en IAIK (TU Graz, Austria).
Ideal para IoT y embebidos: Bajo consumo de energía, estado compacto de 320 bits, sin tablas de lookup ni operaciones costosas.
Una sola pasada: Produce ciphertext + tag en un único recorrido sobre los datos — sin separar el paso de cifrado del de autenticación.
┌─────────────────────────────────────────────────────────┐ │ ASCON-128a — Authenticated Encryption with │ │ Associated Data (AEAD) │ ├─────────────────────────────────────────────────────────┤ │ Inputs : Key (128 bits) + Nonce (128 bits) │ │ + Associated Data (variable, not encrypted) │ │ + Plaintext (variable) │ │ │ │ Outputs : Ciphertext (same length as Plaintext) │ │ + Authentication Tag (128 bits) │ └─────────────────────────────────────────────────────────┘
AEAD: Encrypts the plaintext AND authenticates both the ciphertext and associated data (visible metadata protected against tampering).
NIST LWC Standard 2023: Selected by NIST as the lightweight cryptography standard. Winner of CAESAR 2019 among 57 candidates. Designed at IAIK (TU Graz, Austria).
Ideal for IoT and embedded: Low energy consumption, compact 320-bit state, no lookup tables or expensive operations.
Single pass: Produces ciphertext + tag in a single pass over the data — no separate encryption and authentication steps.

Estado interno: 5 × 64 bits

Internal State: 5 × 64 bits

Estado = [ x0 | x1 | x2 | x3 | x4 ] → 320 bits totales Inicialización (ASCON-128a): x0 = IV = 0x80800c0800000000 ← parámetros del algoritmo x1 = Key_high (bits 127–64 de la clave de 128 bits) x2 = Key_low (bits 63–0 de la clave de 128 bits) x3 = Nonce_high (bits 127–64 del nonce de 128 bits) x4 = Nonce_low (bits 63–0 del nonce de 128 bits) IV decodificada (0x80 80 0c 08): 0x80 = tamaño de clave = 128 bits 0x80 = rate = 128 bits por bloque ← 128a 0x0c = rondas pa = 12 0x08 = rondas pb = 8 ← 128a
Rate (x0‖x1): Los 128 bits superiores interactúan con los datos en cada absorción/extracción. En ASCON-128a el rate es de 128 bits — el doble que ASCON-128.
Capacity (x2‖x3‖x4): Los 192 bits restantes nunca se exponen directamente. Actúan como "memoria secreta" que garantiza la seguridad de la construcción sponge.
Construcción Sponge/Dúplex: El mismo estado de 320 bits absorbe entrada y exprime salida, alternando con la permutación interna según la fase activa.
State = [ x0 | x1 | x2 | x3 | x4 ] → 320 bits total Initialization (ASCON-128a): x0 = IV = 0x80800c0800000000 ← algorithm parameters x1 = Key_high (bits 127–64 of the 128-bit key) x2 = Key_low (bits 63–0 of the 128-bit key) x3 = Nonce_high (bits 127–64 of the 128-bit nonce) x4 = Nonce_low (bits 63–0 of the 128-bit nonce) IV decoded (0x80 80 0c 08): 0x80 = key size = 128 bits 0x80 = rate = 128 bits per block ← 128a 0x0c = pa rounds = 12 0x08 = pb rounds = 8 ← 128a
Rate (x0‖x1): The top 128 bits interact with data during each absorption/extraction. In ASCON-128a the rate is 128 bits — double that of ASCON-128.
Capacity (x2‖x3‖x4): The remaining 192 bits are never directly exposed. They act as "secret memory" that guarantees sponge construction security.
Sponge/Duplex construction: The same 320-bit state absorbs input and squeezes output, alternating with the internal permutation depending on the active phase.

S-box de 5 bits

5-bit S-box

Entrada: columna vertical de 5 bits (1 bit de cada x0–x4) Operación en las 64 columnas simultáneamente: Pre-mezcla: x0 ^= x4; x4 ^= x3; x2 ^= x1 AND-NOT (introduce no-linealidad): t0=x0; t1=x1; t2=x2; t3=x3; t4=x4; x0 = t0 ^ (~t1 & t2); x1 = t1 ^ (~t2 & t3); x2 = t2 ^ (~t3 & t4); x3 = t3 ^ (~t4 & t0); x4 = t4 ^ (~t0 & t1); Post-mezcla: x1 ^= x0; x0 ^= x4; x3 ^= x2; x2 = ~x2
Sin tablas de lookup: Solo AND, NOT y XOR — implementable en hardware con pocos transistores, resistente a timing attacks en software.
No-linealidad: Las operaciones AND hacen imposible describir la S-box con ecuaciones lineales, base del análisis diferencial y lineal.
64 columnas en paralelo: Como opera bit a bit sobre los 5 registros, procesa las 64 posiciones de cada registro simultáneamente — paralelismo masivo en hardware.
Input: 5-bit vertical column (1 bit from each x0–x4) Operation on all 64 columns simultaneously: Pre-mixing: x0 ^= x4; x4 ^= x3; x2 ^= x1 AND-NOT (introduces nonlinearity): t0=x0; t1=x1; t2=x2; t3=x3; t4=x4; x0 = t0 ^ (~t1 & t2); x1 = t1 ^ (~t2 & t3); x2 = t2 ^ (~t3 & t4); x3 = t3 ^ (~t4 & t0); x4 = t4 ^ (~t0 & t1); Post-mixing: x1 ^= x0; x0 ^= x4; x3 ^= x2; x2 = ~x2
No lookup tables: Only AND, NOT and XOR — implementable in hardware with few transistors, resistant to timing attacks in software.
Nonlinearity: AND operations make it impossible to describe the S-box with linear equations — the foundation against differential and linear cryptanalysis.
64 columns in parallel: Operating bit-by-bit over the 5 registers processes all 64 bit positions of each register simultaneously — massive hardware parallelism.

Capa lineal — Difusión

Linear Layer — Diffusion

$$\begin{aligned} x_0 &\leftarrow x_0 \oplus \mathrm{ROR}(x_0,\,19) \oplus \mathrm{ROR}(x_0,\,28) \\ x_1 &\leftarrow x_1 \oplus \mathrm{ROR}(x_1,\,61) \oplus \mathrm{ROR}(x_1,\,39) \\ x_2 &\leftarrow x_2 \oplus \mathrm{ROR}(x_2,\,1) \oplus \mathrm{ROR}(x_2,\,6) \\ x_3 &\leftarrow x_3 \oplus \mathrm{ROR}(x_3,\,10) \oplus \mathrm{ROR}(x_3,\,17) \\ x_4 &\leftarrow x_4 \oplus \mathrm{ROR}(x_4,\,7) \oplus \mathrm{ROR}(x_4,\,41) \end{aligned}$$

Capa lineal — cada registro xi se XOR-ea con 2 rotaciones de sí mismo ROR = rotate right de 64 bits Constantes elegidas para maximizar difusión: x0: (19, 28) x1: (61, 39) x2: (1, 6) x3: (10, 17) x4: (7, 41) Distintas por registro → sin simetría explotable entre xi
Difusión intra-registro: Cada bit de un registro afecta a muchos otros bits del mismo registro tras la rotación y XOR — dispersión completa en pocas rondas.
Difusión inter-registro: La S-box ya creó dependencias entre registros; la capa lineal amplía esas dependencias dentro de cada uno.
Full diffusion: La combinación S-box + capa lineal alcanza full diffusion rápidamente — un solo bit modificado afecta el estado completo en pocas rondas.
Linear layer — each register xi is XORed with 2 rotations of itself ROR = 64-bit rotate right Constants chosen to maximize diffusion: x0: (19, 28) x1: (61, 39) x2: (1, 6) x3: (10, 17) x4: (7, 41) Different per register → no exploitable symmetry between xi
Intra-register diffusion: Each bit of a register affects many other bits in the same register after rotation and XOR — complete spreading in few rounds.
Inter-register diffusion: The S-box already created dependencies between registers; the linear layer amplifies those dependencies within each one.
Full diffusion: The S-box + linear layer combination reaches full diffusion quickly — a single modified bit affects the complete state in few rounds.

Permutaciones pa y pb

Permutations pa and pb

Una ronda ASCON = RC ⊕ x2 → S-box (5 bits) → Capa lineal Constantes de ronda RC (aplicadas a x2 al inicio de cada ronda): r0=0xf0 r1=0xe1 r2=0xd2 r3=0xc3 r4=0xb4 r5=0xa5 r6=0x96 r7=0x87 r8=0x78 r9=0x69 r10=0x5a r11=0x4b pa = 12 rondas (índices 0–11) → Inicialización y Finalización pb = 8 rondas (índices 4–11) → ASCON-128a: AD y Cifrado pb = 6 rondas (índices 6–11) → ASCON-128: AD y Cifrado
ASCON-128 vs ASCON-128a: ASCON-128 usa pb=6 rondas con rate=64 bits por bloque. ASCON-128a usa pb=8 rondas con rate=128 bits — mayor throughput, ligera reducción de margen de seguridad.
RC evitan simetría: Las 12 constantes distintas hacen que cada ronda sea única — evitan slide attacks, puntos fijos y cualquier simetría estructural de la permutación.
pa siempre 12 rondas: Las fases de inicialización y finalización usan siempre pa independientemente de la variante — máximo margen de seguridad en las fases críticas.
One ASCON round = RC ⊕ x2 → S-box (5-bit) → Linear layer Round constants RC (applied to x2 at the start of each round): r0=0xf0 r1=0xe1 r2=0xd2 r3=0xc3 r4=0xb4 r5=0xa5 r6=0x96 r7=0x87 r8=0x78 r9=0x69 r10=0x5a r11=0x4b pa = 12 rounds (indices 0–11) → Initialization and Finalization pb = 8 rounds (indices 4–11) → ASCON-128a: AD and Encryption pb = 6 rounds (indices 6–11) → ASCON-128: AD and Encryption
ASCON-128 vs ASCON-128a: ASCON-128 uses pb=6 rounds with rate=64 bits per block. ASCON-128a uses pb=8 rounds with rate=128 bits — higher throughput, slightly reduced security margin.
RC prevent symmetry: The 12 distinct constants make each round unique — preventing slide attacks, fixed points, and any structural symmetry in the permutation.
pa always 12 rounds: Initialization and finalization phases always use pa regardless of the variant — maximum security margin in the critical phases.

Datos Asociados (AD)

Associated Data (AD)

AD = datos autenticados pero NO cifrados Casos de uso típicos: • Cabeceras de paquetes de red (IP, puerto, protocolo) • Metadatos de mensajes (timestamp, ID de sesión) • Identificadores de dispositivo en IoT/Matter/Thread Proceso de absorción AD (ASCON-128a, rate=128 bits): Para cada bloque de 16 bytes de AD: 1. Padding: bloque ‖ 0x80 ‖ 0x00... hasta 16 bytes 2. (x0‖x1) ⊕= bloque_AD_padded 3. Aplicar pb (8 rondas) Al finalizar todo el AD: x4 ⊕= 1 ← separación de dominio (domain separation)
¿Por qué AD? El receptor verifica que los metadatos no fueron alterados en tránsito, sin necesidad de cifrarlos — eficiencia y flexibilidad en protocolos de red.
Domain separation (x4 ⊕ 1): Garantiza que el estado tras AD vacío sea distinto al estado tras AD no vacío, evitando ambigüedades entre mensajes de diferente estructura.
AD vacío: Si no hay datos asociados, la absorción se omite pero el domain separation (x4 ⊕ 1) se aplica igualmente — el estado siempre avanza de forma determinista.
AD = authenticated but NOT encrypted data Typical use cases: • Network packet headers (IP, port, protocol) • Message metadata (timestamp, session ID) • Device identifiers in IoT/Matter/Thread AD absorption process (ASCON-128a, rate=128 bits): For each 16-byte block of AD: 1. Padding: block ‖ 0x80 ‖ 0x00... up to 16 bytes 2. (x0‖x1) ⊕= padded_AD_block 3. Apply pb (8 rounds) After all AD blocks: x4 ⊕= 1 ← domain separation
Why AD? The receiver verifies that metadata was not altered in transit, without needing to encrypt it — efficiency and flexibility in network protocols.
Domain separation (x4 ⊕ 1): Ensures the state after empty AD is distinct from the state after non-empty AD, avoiding ambiguity between messages of different structure.
Empty AD: If there is no associated data, absorption is skipped but domain separation (x4 ⊕ 1) is applied anyway — the state always advances deterministically.

Las 4 fases del algoritmo

The 4 Algorithm Phases

┌──────────────────────────────────────────────────────────┐ │ 1. INICIALIZACIÓN │ │ Estado = IV ‖ Key ‖ Nonce → pa (12 rondas) │ │ x3 ⊕= Key_high ; x4 ⊕= Key_low │ ├──────────────────────────────────────────────────────────┤ │ 2. ABSORCIÓN AD (si hay datos asociados) │ │ Para cada bloque AD (16 bytes + padding): │ │ (x0‖x1) ⊕= bloque_AD → pb (8 rondas) [128a] │ │ x4 ⊕= 1 (domain separation) │ ├──────────────────────────────────────────────────────────┤ │ 3. CIFRADO │ │ Para cada bloque PT (16 bytes + padding): │ │ CT = (x0‖x1) ⊕ PT (extracción del keystream) │ │ (x0‖x1) = CT (actualizar estado con CT) │ │ → pb (8 rondas) [128a] │ ├──────────────────────────────────────────────────────────┤ │ 4. FINALIZACIÓN │ │ x1 ⊕= Key_high ; x2 ⊕= Key_low │ │ → pa (12 rondas) │ │ Tag = (x3 ⊕ Key_high) ‖ (x4 ⊕ Key_low) [128 bits] │ └──────────────────────────────────────────────────────────┘
Secuencial: Cada fase depende del estado producido por la anterior. La clave aparece en inicialización y finalización — liga el tag a la clave, imposibilitando falsificación.
Estado actualizado con CT: En la fase de cifrado el estado absorbe el ciphertext (no el plaintext). Esto garantiza que el tag cubre el CT y no solo el PT.
┌──────────────────────────────────────────────────────────┐ │ 1. INITIALIZATION │ │ State = IV ‖ Key ‖ Nonce → pa (12 rounds) │ │ x3 ⊕= Key_high ; x4 ⊕= Key_low │ ├──────────────────────────────────────────────────────────┤ │ 2. AD ABSORPTION (if there is associated data) │ │ For each AD block (16 bytes + padding): │ │ (x0‖x1) ⊕= AD_block → pb (8 rounds) [128a] │ │ x4 ⊕= 1 (domain separation) │ ├──────────────────────────────────────────────────────────┤ │ 3. ENCRYPTION │ │ For each PT block (16 bytes + padding): │ │ CT = (x0‖x1) ⊕ PT (keystream extraction) │ │ (x0‖x1) = CT (update state with CT) │ │ → pb (8 rounds) [128a] │ ├──────────────────────────────────────────────────────────┤ │ 4. FINALIZATION │ │ x1 ⊕= Key_high ; x2 ⊕= Key_low │ │ → pa (12 rounds) │ │ Tag = (x3 ⊕ Key_high) ‖ (x4 ⊕ Key_low) [128 bits] │ └──────────────────────────────────────────────────────────┘
Sequential: Each phase depends on the state produced by the previous one. The key appears in initialization and finalization — binding the tag to the key, making forgery impossible.
State updated with CT: In the encryption phase, the state absorbs the ciphertext (not the plaintext). This ensures the tag covers the CT and not just the PT.

Tag de autenticación y seguridad

Authentication Tag and Security

Tag (128 bits) = (x3 ⊕ Key_high) ‖ (x4 ⊕ Key_low) Para descifrar y verificar: 1. Recorrer el proceso completo con Key, Nonce, AD, CT 2. Calcular tag_calculado 3. Comparar en tiempo constante: tag_calculado == tag_recibido ✓ Iguales → mensaje auténtico e íntegro → aceptar PT ✗ Distintos → mensaje alterado o clave errónea → rechazar Propiedades formales de seguridad: Confidencialidad : IND-CCA2 (128 bits de seguridad) Integridad/Auth : INT-CTXT (128 bits de seguridad) Nonce : único por (Key, mensaje) — obligatorio
Nonce misuse: Si se reutiliza el mismo nonce con la misma clave, un atacante puede hacer CT1 ⊕ CT2 = PT1 ⊕ PT2 y recuperar los plaintexts. El nonce DEBE ser único.
Comparación en tiempo constante: Nunca compares tags byte a byte con early exit — un atacante puede medir el tiempo y deducir cuántos bytes coinciden (timing attack).
Implementación de producción: Para uso real, usá una librería auditada: libsodium, BouncyCastle, mbedTLS (con ASCON support). Esta visualización es educativa.
Tag (128 bits) = (x3 ⊕ Key_high) ‖ (x4 ⊕ Key_low) To decrypt and verify: 1. Run the full process with Key, Nonce, AD, CT 2. Compute computed_tag 3. Constant-time compare: computed_tag == received_tag ✓ Match → message is authentic and intact → accept PT ✗ Mismatch → message altered or wrong key → reject Formal security properties: Confidentiality : IND-CCA2 (128-bit security) Integrity/Auth : INT-CTXT (128-bit security) Nonce : unique per (Key, message) — mandatory
Nonce misuse: If the same nonce is reused with the same key, an attacker can compute CT1 ⊕ CT2 = PT1 ⊕ PT2 and recover both plaintexts. The nonce MUST be unique.
Constant-time comparison: Never compare tags byte by byte with early exit — an attacker can measure timing and deduce how many bytes match (timing attack).
Production use: For real use, use an audited library: libsodium, BouncyCastle, mbedTLS (with ASCON support). This visualizer is for educational purposes.

¿Cuándo usar 128a? Cuando el throughput importa (mensajes largos, dispositivos con CPU moderada). Rate doble = el doble de datos procesados por invocación de pb.
¿Cuándo usar 128? Cuando el margen de seguridad es prioritario sobre la velocidad, o en hardware extremadamente restringido donde el rate pequeño ayuda al diseño.

When to use 128a? When throughput matters (long messages, moderately-powered devices). Double rate = twice as much data processed per pb invocation.

ASCON en el mundo real — SOC / DFIR

ASCON in the Real World — SOC / DFIR

Identificar ASCON en binarios y firmware
  • Firma de ASCON-128a: buscar 80 80 0C 08 00 00 00 00 en big-endian — esa es la IV que inicializa el estado. Su presencia indica una implementación de ASCON-128a.
  • Firma de ASCON-128: buscar 80 40 0C 06 00 00 00 00 — IV de ASCON-128.
  • Comando grep: grep -rl $'\x80\x80\x0c\x08' firmware/ o con binwalk --signature si tenés reglas customizadas.
  • YARA rule: uint64be(offset) == 0x80800c0800000000 como condición para detectar el estado inicial.
Dónde encontrar ASCON en producción: Protocolos IoT : Matter, Thread, OSCORE (CoAP/6LoWPAN) Firmware : STM32 + libsodium, ESP32-C3, nRF52840 Librerías : ascon-c (C), ascon-rs (Rust), pyascon (Python) TLS : DTLS 1.3 con cipher suite TLS_ASCON_128A_SHA256 Escenario de ataque — nonce misuse en producción: Si un dispositivo IoT reinicia y resetea su contador de nonces: • El nonce se repite con la misma clave • CT1 ⊕ CT2 = PT1 ⊕ PT2 (crib-dragging posible) • Con palabras conocidas del protocolo → PT recuperable Mitigación: nonce con timestamp de alta resolución o DRBG
Vector de análisis en DFIR: Al analizar tráfico cifrado en IoT, identificar la variante ASCON (por la IV) permite determinar el tamaño de rate y los límites de seguridad de cada mensaje.
Hardening: En audit de firmware, verificar que los nonces no se generen con PRNG sin semilla post-reset. Un generador de nonces predecible rompe la seguridad AEAD.
