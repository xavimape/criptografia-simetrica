¿Qué es una función hash criptográfica?

Una función hash criptográfica toma una entrada de longitud arbitraria y produce una salida de longitud fija llamada digest o resumen.

Tres propiedades fundamentales de seguridad:

  • Resistencia a preimagen: dado H, no es factible encontrar M tal que hash(M) = H.
  • Resistencia a segunda preimagen: dado M1, no es factible encontrar M2 ≠ M1 con el mismo hash.
  • Resistencia a colisiones: no es factible encontrar M1 ≠ M2 tal que hash(M1) = hash(M2).
Mensaje (longitud variable) "hello world" → b94d27b9... (SHA-256, 256 bits) "Hello world" → a4f17b20... (SHA-256, 256 bits) archivo de 4 GB → 3e23e816... (SHA-256, 256 bits) ↕ 1 bit de diferencia → ~50% de bits del digest distintos (efecto avalancha)
🔁 Una sola dirección: es computacionalmente inviable revertir un hash — no existe función inversa eficiente.
📐 Longitud fija: sin importar el tamaño de la entrada, el digest tiene siempre la misma longitud (SHA-256 → 256 bits, SHA-3-256 → 256 bits).
02 / 08

Construcción Merkle-Damgård

La mayoría de las funciones hash clásicas (MD5, SHA-1, SHA-2) usan la construcción iterativa Merkle-Damgård: se divide el mensaje en bloques de tamaño fijo, se aplica una función de compresión a cada bloque junto con el estado anterior, y el resultado final es el hash.

Vulnerabilidad inherente: la construcción Merkle-Damgård es susceptible a ataques de extensión de longitud (length extension attack). SHA-3 y HMAC resuelven este problema.

Mensaje + padding ┌──────┬──────┬──────┬──────┐ │ B0 │ B1 │ B2 │ B3 │ bloques de 512 bits └──────┴──────┴──────┴──────┘ │ │ │ │ H0─┤ H1──┤ H2──┤ H3──┤ f() f() f() f() └── digest final
⚠️ Length extension attack: conocido hash(M) y len(M), es posible calcular hash(M ‖ padding ‖ M') sin conocer M. Afecta a MD5, SHA-1 y SHA-2 (no a SHA-3).
03 / 08

MD5 — roto pero ubicuo

MD5 (1991) produce un digest de 128 bits (32 caracteres hex). Opera sobre bloques de 512 bits con un estado de cuatro palabras de 32 bits (A, B, C, D). Aplica 4 rondas de 16 operaciones cada una usando funciones lógicas F, G, H, I.

  • 2004: Wang & Yu demuestran colisiones analíticas.
  • 2012: malware Flame usa colisiones MD5 para falsificar certificados de Microsoft.
  • Hoy: colisiones en segundos en hardware común. No usar para seguridad.

Aún aparece en: checksums de descargas no críticas, sistemas legado, y como identificador de ficheros en plataformas de threat intelligence (MD5 de malware en VirusTotal).

Ejemplo de colision MD5: file1.bin → 79054025255fb1a26e4bc422aef54eb4 (idéntico!) file2.bin → 79054025255fb1a26e4bc422aef54eb4 (contenido diferente)
04 / 08

SHA-1 — obsoleto, pero presente en infraestructura

SHA-1 (1995, FIPS 180-1) produce un digest de 160 bits (40 hex). Merkle-Damgård con bloques de 512 bits, 80 rondas, estado de 5 palabras de 32 bits.

  • 2005: Wang et al. reducen la complejidad de colisión a 269 (teórico).
  • 2017: Google & CWI demuestran colisión real — SHAttered — dos PDFs distintos con el mismo SHA-1.
  • NIST SP 800-131A: prohibido para firmas digitales desde 2013.

Detectar SHA-1 en configuraciones TLS (p.ej. TLS_RSA_WITH_AES_128_CBC_SHA) es una señal de configuración débil. Los certificados SHA-1 fueron eliminados por navegadores en 2017. Git usa SHA-1 para identificar commits (migración a SHA-256 en curso).

SHA-1 en cipher suite TLS (indicador de debilidad): TLS_RSA_WITH_AES_128_CBC_SHA ← SHA-1 como HMAC TLS_RSA_WITH_AES_256_CBC_SHA ← SHA-1 como HMAC TLS_ECDHE_RSA_WITH_AES_256_CBC_SHA384 ← SHA-384 (OK)
05 / 08

SHA-256 — el estándar operativo actual

SHA-256 (SHA-2, FIPS 180-4) produce un digest de 256 bits (64 hex). Merkle-Damgård con bloques de 512 bits, 64 rondas, ocho palabras de 32 bits en el estado. Las constantes K[0..63] se derivan de las raíces cúbicas de los primeros 64 números primos.

En cada ronda se aplican:

  • Ch(e,f,g): \(Ch(e,f,g) = (e \wedge f) \oplus (\lnot e \wedge g)\)
  • Maj(a,b,c): \(Maj(a,b,c) = (a \wedge b) \oplus (a \wedge c) \oplus (b \wedge c)\)
  • \(\Sigma_0(a) = ROTR^{2}(a) \oplus ROTR^{13}(a) \oplus ROTR^{22}(a)\)
  • \(\Sigma_1(e) = ROTR^{6}(e) \oplus ROTR^{11}(e) \oplus ROTR^{25}(e)\)
SHA-256 en uso real: TLS 1.3 cipher suite: TLS_AES_256_GCM_SHA384 JWT header: {"alg":"HS256"} Bitcoin mining: SHA-256(SHA-256(block_header)) Malware IOC (TI): sha256: 3e23e8160d2b... DFIR cadena custodia: sha256sum evidence.dd > evidence.sha256
06 / 08

SHA-3 / Keccak — construcción esponja

SHA-3 (FIPS 202, 2015) es el ganador del concurso NIST SHA-3 Competition (2007-2012). A diferencia de MD5/SHA-1/SHA-2, no usa la construcción Merkle-Damgård, sino una construcción esponja (sponge function) sobre un estado de 1600 bits.

Ventajas sobre SHA-2: resistente a length extension attacks por diseño, y ofrece variantes como SHAKE128/SHAKE256 (XOF: salida de longitud variable) y cSHAKE para derivación de claves customizada.

Estado Keccak: 5×5 matriz de lanes de 64 bits (1600 bits total) FASE ABSORCION: Mensaje (en bloques de rate=136 bytes para SHA3-256) → XOR en los primeros 136 bytes del estado → Aplicar permutacion Keccak-f (24 rondas: θ ρ π χ ι) → Repetir para cada bloque FASE EXPRIMIDO (SQUEEZE): Extraer los primeros 256 bits del estado → digest

HMAC — autenticación con hash

HMAC (Hash-based Message Authentication Code, RFC 2104) combina una clave secreta con una función hash para producir un MAC que autentica tanto el origen como la integridad del mensaje.

$$HMAC(K, M) = H\!\Bigl((K \oplus opad) \;\Big\|\; H\!\bigl((K \oplus ipad) \;\|\; M\bigr)\Bigr)$$

ipad = 0x36 repetido blocksize veces opad = 0x5C repetido blocksize veces 1. Si len(K) > blocksize : K = H(K) 2. Si len(K) < blocksize : K = K ‖ 0x00... 3. inner = H( (K ⊕ ipad) ‖ M ) 4. HMAC = H( (K ⊕ opad) ‖ inner )
🔑 JWT (HS256): firma de tokens con HMAC-SHA256. El servidor verifica la firma antes de confiar en el payload.
🔐 TLS 1.2 PRF y TLS 1.3 HKDF: HMAC es la base del KDF de dos fases (Extract+Expand) que genera las claves de sesión.
🌐 API authentication: AWS SigV4, webhooks de GitHub y Stripe usan HMAC-SHA256 para firmar cada solicitud.
08 / 09

Hashing en ciberseguridad operacional

Más allá de la teoría, los hashes tienen un rol central en operaciones diarias del SOC: desde verificar integridad de evidencia digital hasta identificar malware en plataformas de threat intelligence.

── IOCs en Threat Intelligence ───────────────────────────── VirusTotal: buscar por SHA-256 → variantes, familias, TTPs MISP: atributo tipo 'sha256' en eventos de malware OpenCTI: pivot desde hash → campaign → actor group MalwareBazaar: SHA-256 como clave primaria de muestra ── DFIR — Cadena de Custodia ──────────────────────────────── sha256sum evidence.dd > evidence.sha256 # antes de analizar FTK Imager → calcula MD5 + SHA-1 + SHA-256 automáticamente Autopsy → hash de imagen + archivos extraídos Un hash distinto al documentado INVALIDA la evidencia ── Contraseñas y autenticación ────────────────────────────── NTLM hash = MD4(UTF-16LE(password)) ← Windows SAM/AD bcrypt/scrypt/Argon2 ← hashes modernos con sal y costo hashcat --hash-type 1000 hashes.txt wordlist.txt
🎯 Reglas YARA con hash: hash.sha256(0, filesize) en condiciones YARA para detección exacta por hash. Combinado con strings para variantes: detecta familia aunque cambien bytes individuales.
📋 Sigma para SIEM: campo Hashes|contains en reglas Sigma sobre eventos Sysmon (Event ID 1 - Process Create). Un EDR que reporta hash SHA-256 de proceso permite correlación con TI feeds.
🛡️ Imphash y ssdeep: el import hash (MD5 de la tabla de imports PE) identifica compilaciones del mismo código. ssdeep (fuzzy hash) encuentra variantes con diferencias menores al 30%.

YARA, Sigma y hash pivoting en threat intel

El trabajo operacional con hashes involucra reglas automatizadas, pivoting entre plataformas y detección de evasión — habilidades core para analistas SOC Tier 1 y 2.

── Workflow de hash pivoting ──────────────────────────────── 1. Muestra → sha256sum sample.exe 2. Hash SHA-256 → VirusTotal: detecciones, familias, TTPs 3. VT → "Similar files" → otras muestras de la campaña 4. Hash → MISP: correlación con eventos de incidentes previos 5. Imphash → variantes compiladas del mismo código fuente ── Regla YARA con hash ────────────────────────────────────── import "hash" rule Ransomware_Exact { condition: hash.sha256(0, filesize) == "44d88612fea8a8f36de82e1278abb02f" "c9d2ef1b5b25aa5c7e78c45c7b420b7e" } ── Regla Sigma (Sysmon Event ID 1) ───────────────────────── detection: selection: EventID: 1 Hashes|contains: '44D88612FEA8A8F36DE82E1278ABB02F' condition: selection ── Evasión a detectar ─────────────────────────────────────── Timestomping → $SI.Modified ≠ $FN.Modified (MFT) Packing → cambia hash pero imphash puede persistir Section rename → mismo código, diferente layout PE
🎯 Hash pivoting: un solo hash SHA-256 de muestra permite encontrar variantes, campañas y actores en VirusTotal, MISP y feeds de TI. El pivoting por imphash es especialmente efectivo para variantes compiladas del mismo código.
🔐 Forensia de contraseñas: dumps SAM/NTDS contienen NT hashes (MD4 de la contraseña Unicode). Herramientas: Mimikatz (lsadump::sam), Secretsdump. Cracking offline con hashcat. Detectar acceso via Sysmon Event ID 10 (LSASS).
📋 Cadena de custodia digital: SHA-256 del dispositivo completo (sha256sum /dev/sda), cada partición y cada archivo extraído. En sede judicial, 1 bit de diferencia invalida la evidencia. FTK Imager y Autopsy generan reportes de hash automáticamente.
